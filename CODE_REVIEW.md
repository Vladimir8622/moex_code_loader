# Код-ревью MOEX Data Loader

Ниже — находки по каждому файлу. Критичные баги (ломают пайплайн на реальных
данных) отмечены 🔴, остальное — 🟡 качество/надёжность, 🔵 косметика.
Все 🔴 и часть 🟡 закрыты тестами в `tests/` (см. `TESTS_README.md`).

## 🔴 Критичные баги

### 1. `load_summary.py` — `log(0)` роняет весь скрипт почти всегда
```python
markersize=0.5 * np.log(row['file_size'])
```
Строится для **каждой** комбинации (тикер, год, месяц), включая те, где
файла нет вообще (`file_size = 0`) — а это нормальная ситуация: далеко не
каждый тикер торгуется во всех 12 буквах месяцев. `log(0) = -inf`,
matplotlib не может отрисовать точку с такой markersize и падает с
`SystemError: tile cannot extend outside image`. На реальных данных это
почти гарантированно произойдёт → `update_data.py` остановится на шаге 2 и
никогда не дойдёт до склейки контрактов (шаг 3).
**Тест:** `test_pipeline_crashes_on_missing_contract_files_bug`.
**Фикс:** отфильтровать `file_size == 0` перед циклом отрисовки
(`df_ticker[df_ticker['file_size'] > 0]`) — проверено тестом
`test_pipeline_succeeds_once_zero_size_rows_are_filtered`.

### 2. `load_summary.py` — нулевой `figsize` при малом числе тикеров
```python
plt.figure(figsize=(int(len(series_names)/4), int(len(tickers)/3)))
```
Целочисленное деление: при 1–2 тикерах в категории (вполне вероятно —
не каждый ticker торгует все контракты) высота фигуры округляется до 0,
и `savefig()` падает независимо от бага №1.
**Тест:** `test_figsize_zero_bug_with_few_tickers`.
**Фикс:** `figsize=(max(1, len(series_names)//4), max(1, len(tickers)//3))`.

### 3. `utilities.get_futures_active_tickers()` — `KeyError` на пустом ответе биржи
```python
tickers = pd.DataFrame(tickers)['ticker'].tolist()
```
Если API вернул пустой список (сбой/выходной день/пустой рынок),
`pd.DataFrame([])` не имеет колонки `'ticker'` → `KeyError` вместо
graceful empty list. Роняет весь пайплайн из-за проблемы на стороне биржи.
Дублируется в `moex_futures_merge.get_futures_active_tickers()`.
**Тест:** `test_empty_market_raises_keyerror_bug`.
**Фикс:** `if not tickers: return []` перед созданием DataFrame.

### 4. `load_summary.py` — мёртвый код, который упадёт при вызове
В конце файла, **после** всех вызовов `summary_plot(...)`, повторно
определяется:
```python
def generate_names():
    ...
    tickers = get_futures_active_tickers()   # <-- не импортирован в этом файле!
    ...
```
Функция никогда не вызывается (мёртвый код), но если её вызвать —
получите `NameError`, т.к. `get_futures_active_tickers` в этом модуле не
определена и не импортирована (есть только `utilities.get_futures_active_tickers`).
Похоже на случайно оставленный дубликат `utilities.generate_names`.
**Рекомендация:** удалить.

## 🟡 Надёжность / логика

### 5. TODO не закрыт: тикеры длиннее 2 символов обрабатываются неверно
`utilities.get_futures_active_tickers()` делает `ticker[:-2]`, то есть
жёстко предполагает формат `КОРЕНЬ(любой длины) + МЕСЯЦ(1) + ГОД(1)`, что
формально верно, НО `load_summary.py` при классификации на
monthly/quarterly режет **по фиксированной позиции**:
```python
month_names_in_files = sorted(set([file[2:3] for file in ticker_files]))
```
Это предполагает, что корень тикера — ровно 2 символа. Для тикеров с
другой длиной корня буква месяца извлечётся неверно → тикер попадёт не в
ту категорию (monthly/quarterly) молча, без ошибки. Ровно то, о чём
предупреждает `# TODO: tickers with more than two symbols` в
`moex_futures_data_loader.py` — но TODO не выполнено, и баг живёт в другом
файле, где его сложнее заметить.
**Тест:** `test_three_letter_root_is_mishandled` (документирует текущее
поведение как якорь для будущего фикса).

### 6. Дублирование кода между `utilities.py` и `moex_futures_merge.py`
`get_futures_active_tickers()` и `generate_names()` определены дважды
дословно одинаково (в `utilities.py`, и второй раз — в
`moex_futures_merge.py`, хотя этот файл уже импортирует `utilities` и
даже не использует этот импорт). Риск: правки/фиксы в одном месте не
попадут во второе → расхождение логики. **Рекомендация:** удалить
дубликаты в `moex_futures_merge.py`, вызывать `utilities.generate_names()`.

### 7. `load_summary.py` выполняется целиком на уровне модуля (нет `if __name__ == '__main__':`)
Любой `import load_summary` (например, из теста) немедленно скачивает
список тикеров, читает диск и рисует графики. Это не даёт использовать
код повторно/тестировать без побочных эффектов и запуска полного
пайплайна. Также `summary_plot()` неявно берёт `years` из глобальной
области видимости модуля вместо параметра — работает только благодаря
случайному порядку объявлений в файле.

### 8. `moex_futures_merge.py` — потенциальный `IndexError` в определении первого ролловера
```python
rollover_date = volumes.index[(volumes.iloc[:,0] > 0.5*np.quantile(...))][0]
```
Если такой день не найден (пустой массив), `[0]` упадёт с `IndexError`.
Также используются квантили (`np.quantile(..., 0.95)`), которые при малом
числе точек/низкой ликвидности могут выродиться в 0 — тогда условия
фильтрации становятся тривиальными и ролловер может определиться
слишком рано/на шуме. Стоит добавить проверки на пустые срезы и
минимальный объём данных.

### 9. `moex_futures_data_loader.make_rets()` — недостижимый код
```python
    # ... for-loop с return в каждой ветке ...
    time.sleep(random.uniform(5, 10))
    return False
```
Эти две строки после `for`-цикла никогда не выполняются — каждая ветвь
внутри цикла уже делает `return`. Мёртвый код, вводящий в заблуждение
(похоже на попытку сделать паузу между *успешными* запросами, но эта
пауза не срабатывает никогда).

### 10. `update_data.py` — вывод дочерних процессов не стримится
`subprocess.run(..., capture_output=True)` подавляет весь `stdout`
(включая прогресс-бары `tqdm` и `print`) до завершения процесса. Для
многочасовой загрузки данных это неудобно — не видно прогресса, только
итог. Стоит убрать `capture_output` (или использовать `Popen` с
построчным чтением) для длительных шагов.

## 🔵 Мелочи

- Файл `_gitignore` в репозитории называется без ведущей точки — для
  git это обычный файл, а не `.gitignore`, поэтому исключения (`MOEX/`,
  `continous/`, `summary/`, `venv/` и т.д.) **не применяются**. Нужно
  переименовать в `.gitignore`.
- Опечатка "continous" вместо "continuous" — используется последовательно
  и в коде, и в `.gitignore`, так что менять безопасно, но стоит иметь в
  виду при поиске по кодовой базе.
- Двойной `import pandas as pd` в `moex_futures_data_loader.py` и
  `load_summary.py` (импортируется дважды в одном файле).
- `requirements.txt` содержит только нижние границы версий
  (`pandas>=1.5.0` и т.д.) без верхних — в песочнице с `pandas 3.0`
  код уже даёт `Pandas4Warning: 'd' is deprecated, please use 'D'
  instead` на `resample('1d')`. Стоит либо запинить версии, либо сразу
  поправить `'1d'` → `'D'`, `'5min'` → `'5min'` (уже ок) и т.п.

## Итог по работоспособности
Скрипты **синтаксически корректны** и импортируются без ошибок. Однако
`load_summary.py` в реалистичном сценарии (не все тикеры торгуют все
контракты — а так почти всегда) **упадёт** из-за багов №1 и №2, оборвав
весь пайплайн `update_data.py` на втором шаге. Ядро логики
(`moex_futures_merge.merge_futures`, ролловер по объёму) в проверенных
тестами сценариях работает корректно: контракт с падающим объёмом
корректно сменяется контрактом с растущим объёмом, дата ролловера
записывается, итоговый ряд склеен без пропусков и без смешивания
тикеров по разные стороны от даты ролловера.
