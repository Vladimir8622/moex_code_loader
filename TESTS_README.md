# Тесты

Тесты используют только стандартный `unittest` (без pytest — в среде без
доступа к сети не было возможности его установить, но тесты в pytest
запустятся точно так же благодаря совместимому API).

Сеть недоступна и не нужна: `apimoex`, `moexalgo`, `tqdm` подменены
лёгкими заглушками в `tests/stubs/` (это НЕ рабочие клиенты MOEX —
только интерфейс, достаточный чтобы импортировать и тестировать реальный
код без сетевых вызовов).

`load_summary.py` тестами не покрывается (по решению автора).

## Запуск всех тестов
```bash
cd moex_data_loader
PYTHONPATH=. python3 -m unittest discover -s tests -v
```
(или `pytest tests/ -v`, если pytest установлен). Путь до `tests/stubs/`
каждый тест-файл добавляет в `sys.path` сам (см. `sys.path.insert(...)` в
начале файла), поэтому в `PYTHONPATH` достаточно указать только корень
проекта (`.`).

## Что покрыто

| Файл | Что проверяется |
|---|---|
| `tests/test_utilities.py` | Формирование корня тикера (`ticker[:-2]`) и дедупликация в `get_futures_active_tickers()` |
| `tests/test_data_loader.py` | `make_rets()`: запись CSV с `log_ret`, поведение на пустом ответе API, retry/backoff на сетевых ошибках, отсутствие retry на прочих исключениях |
| `tests/test_merge_algorithm.py` | Алгоритм ролловера в `merge_futures()`: переход с затухающего фронт-контракта на растущий по объёму следующий контракт происходит только когда второй контракт действительно стал ликвиднее; итоговый ряд хронологически монотонен и не смешивает тикеры по разные стороны от даты ролловера; для одного контракта ролловеров не бывает; для тикера без данных не создаётся файлов; несовпадение `years` в конфиге с реальными данными молча даёт пустой результат; ресемплинг 1-мин → 5-мин в `merge_5min()` |

## Архитектурные изменения, повлиявшие на тесты

- `make_rets()` теперь принимает `data_folder` как обязательный
  позиционный аргумент (`pathlib.Path`) вместо чтения модульной
  переменной `loader.data_folder`. Тесты обновлены, чтобы передавать
  путь явно, и мокают `time.sleep` во всех сценариях (в т.ч. успешных),
  т.к. на успехе код теперь реально ждёт `random.uniform(2,3)` секунд.
- `merge_futures()` теперь принимает `config` (словарь, как из
  `config.yaml`, с ключами `config['general']['months']` /
  `config['general']['years']`) вместо неявного использования генератора
  имён. Тесты собирают минимальный `config` под каждый сценарий.
- `utilities.generate_names()` больше не существует — тесты на неё
  удалены.

## Логика проверки корректности алгоритма склейки (moex_futures_merge)

Вместо проверки "ролловер произошёл ровно такого-то числа" (что зависело
бы от деталей квантильной эвристики и было бы хрупким), тесты проверяют
**инварианты**, которым обязан удовлетворять любой корректный непрерывный
ряд:

1. ролловер происходит не раньше, чем задний контракт реально становится
   более ликвидным, чем передний;
2. итоговый индекс дат монотонно не убывает (годится как ценовой ряд);
3. все строки до даты ролловера — из первого контракта, все строки после
   — из второго (без "протечек" данных через границу);
4. для единственного контракта без данных о втором — ролловеров нет и
   выходной ряд целиком состоит из первого контракта;
5. для тикера без единого файла — не создаётся ни `_rolls.csv`, ни
   `_1min.csv`;
6. несовпадение `years` в конфиге с реальными данными на диске молча
   даёт пустой результат вместо ошибки — задокументировано отдельным
   тестом как поведение при неправильной настройке конфига.

## Известное ограничение самих тестов

Алгоритм ролловера в `merge_futures()` имеет потенциальную логическую
проблему, которую текущие тесты **не покрывают**: `volumes_prev`
обновляется только когда ролловер найден на данной итерации. Если между
двумя соседними месяцами ролловер не находится (например, контракт
слишком неликвиден), `volumes_prev` не обновляется, и следующий контракт
в цепочке сравнивается не с предыдущим, а с контрактом через один —
целый контракт может выпасть из цепочки склейки. Стоит покрыть отдельным
тестом (3+ последовательных контракта, один из которых пропускает явный
момент ролловера), если такой сценарий реален для ваших данных.
