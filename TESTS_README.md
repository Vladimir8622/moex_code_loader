# Тесты

Тесты используют только стандартный `unittest` (без pytest — в среде без
доступа к сети не было возможности его установить, но тесты в pytest
запустятся точно так же благодаря совместимому API).

Сеть недоступна и не нужна: `apimoex`, `moexalgo`, `tqdm` подменены
лёгкими заглушками в `stubs/` (это НЕ рабочие клиенты MOEX — только
интерфейс, достаточный чтобы импортировать и тестировать реальный код
без сетевых вызовов).

## Запуск всех тестов
```bash
cd moex_data_loader
PYTHONPATH=stubs:. python3 -m unittest discover -s tests -v
```
(или `pytest tests/ -v`, если pytest установлен — `sys.path` уже
настраивается внутри каждого тест-файла).

## Что покрыто

| Файл | Что проверяется |
|---|---|
| `tests/test_utilities.py` | Формирование корня тикера (`ticker[:-2]`), дедупликация, форма `generate_names()`; регрессионные тесты на найденные баги (пустой ответ биржи → `KeyError`, некорректная обработка тикеров с корнем длиннее 2 символов) |
| `tests/test_data_loader.py` | `make_rets()`: запись CSV с `log_ret`, поведение на пустом ответе API, retry/backoff на сетевых ошибках, отсутствие retry на прочих исключениях |
| `tests/test_merge_algorithm.py` | Ключевая бизнес-логика — алгоритм ролловера в `merge_futures()`: переход с затухающего фронт-контракта на растущий по объёму следующий контракт происходит **только** когда второй контракт действительно стал ликвиднее; итоговый ряд хронологически монотонен и не смешивает тикеры по разные стороны от даты ролловера; для одного контракта ролловеров не бывает; для тикера без данных не создаётся файлов |
| `tests/test_load_summary.py` | Сквозной (subprocess) прогон `load_summary.py`: фиксирует два найденных краш-бага (`log(0)` в markersize и нулевой `figsize`) как регрессионные тесты и подтверждает, что предложенный однострочный фикс действительно решает проблему |

## Логика проверки корректности алгоритма склейки (moex_futures_merge)

Вместо проверки "ролловер произошёл ровно такого-то числа" (что зависело
бы от деталей квантильной эвристики и было бы хрупким), тесты проверяют
**инварианты**, которым обязан удовлетворять любой корректный непрерывный
ряд:

1. ролловер происходит не раньше, чем задний контракт реально становится
   более ликвидным, чем передний;
2. итоговый индекс дат монотонно не убывает (годится как ценовой ряд);
3. все строки до даты ролловера — из первого контракта, все строки после
   — из второго (без "протечек" данных через границу);
4. для единственного контракта без данных о втором — ролловеров нет и
   выходной ряд целиком состоит из первого контракта;
5. для тикера без единого файла — не создаётся ни `_rolls.csv`, ни
   `_1min.csv`.
